# SmartRobotArm v1 — 开发进度与待办事项

> **8 周冲刺计划**：从原本的 mobile manipulation + garment sorting 路线，转向**固定式桌面操作工作站 (fixed tabletop manipulation workstation)**，专注 ANIMA 认知层 + ACT 低层技能的全栈具身智能展示。
>
> 本文档是 v1 实机开发的唯一执行真源（single source of truth）。设计文档（`SOMA_HOME_EXP/`）已于 2026-04-07 同步更新；本仓库 `SmartRobotArm` 由 [`soma_home_exp_v1`](https://github.com/jeffliulab/soma_home_exp_v1)（已归档）迁移而来。

## 本次更新摘要 (Changes from previous version)

**2026-04-07 重大转向**：放弃"自制底盘 + 跨房间衣物分拣"的 29 周路线，改为 **8 周固定桌面工作站冲刺**——单臂 RoArm-M2-S 固定在小桌一侧，头顶固定相机俯视 A4 工作区，做 language-conditioned pick-and-place。原有的 mobile manipulation 故事**仅在 Gazebo 仿真中保留**（用于"system integration"叙事），实机硬件 demo 全部为固定站。

**为什么转向**：4-DOF 单臂 + 廉价舵机做不了 garment folding（需要 ALOHA 级双臂 + 力反馈），mobile + Nav2 + SLAM 集成需要数月且无法突出认知层，行业里 ALOHA / RT-1 / OpenVLA / LeRobot 全部从固定站起步——这是市场识别度最高的范式。

**2026-04-07 硬件 + 开发环境锁定**：相机定为 **Logitech C922 Pro** (~$80, USB UVC, ALOHA 同款)，开发环境定为 **WSL2 + Ubuntu 22.04 + CUDA passthrough**（本机 Linux 双系统 GPU 性能受限，WSL2 直接用 Windows NVIDIA driver 满血），桌面物体改为**小玩偶 / 小公仔**（用户已有，比积木更"宽容"机械臂的低精度抓取）。Pi 5 在 v1 真机上的角色待定——USB 相机直插 PC 后，Pi 5 在 v1 真机几乎没有作用，可能下放给 v2 嵌入式部署阶段。

## 战略转向

### 旧计划（已废弃）
- Mobile manipulation：自制底盘 + RoArm-M2-S 跨房间执行 garment sorting + cable management
- 主任务："把脏衬衫放进洗衣篮"
- ACT 在 mobile platform 上从 teleop 数据训练
- 时间预算：8 个月，11 个 Phase

### 旧计划为什么走不通

| 问题 | 解释 |
|---|---|
| **硬件能力错配** | 4-DOF 单臂 + STS3215 hobby 舵机无法做衣物折叠/分拣——这类任务需要 ALOHA 那种双臂 + 力反馈 + 高精度。坚持做下去就是"用钳子做心脏手术"。 |
| **集成成本压垮认知层** | 自制底盘 + Nav2 + SLAM + 多机通信 + sim2real 调参，每一项都是数月工程，做完后剩下能演示 ANIMA 的时间几乎为零。 |
| **行业范式不对** | ALOHA、RT-1、OpenVLA、LeRobot 官方教程**全部从固定工作站 (fixed station) 起步**。Mobile ALOHA 是后来的扩展，不是入门姿势。HR 一眼能识别的是 ALOHA-style 范式，不是自制底盘。 |
| **Gazebo 不支持软体** | 衣物操作需要 cloth physics，Gazebo 完全做不了。要做必须上 Isaac Sim，又是一个月的工具链学习。 |
| **求职窗口压力** | 项目存在的目的是产出可演示的 portfolio，不是做研究。所有不能在 8 周内拍成 demo 视频的功能都要砍。 |

### 新计划核心思想

> **ANIMA 认知层是这个项目的主角，ACT 是它的低层 API，硬件只是一个能让 demo 有"真机"二字的载体。**
>
> 把所有工程预算压在能让"自然语言 → 真实机械臂动作 → 视觉验证 → 失败恢复"这条 demo 视频成立的事情上。其它一切——mobile、衣物、双臂、力控——全部砍掉或推迟到 v2。

### 行业参照

| 项目 | 起步形态 | 结论 |
|---|---|---|
| ALOHA (Stanford) | 固定双臂工作站 | Mobile ALOHA 是 18 个月后才出的扩展 |
| RT-1 / RT-2 (Google) | 固定 Everyday Robot 桌面 | 全部固定，没有 mobile pick |
| OpenVLA (Stanford/TRI) | 固定 WidowX 桌面 | 教程标配 |
| LeRobot (HuggingFace) | 固定 SO-100 / SO-101 | 官方文档全部桌面 |
| π0 / π0.5 (Physical Intelligence) | 固定双臂 ALOHA / DROID | 真机评测全是固定场 |

**结论**：固定桌面工作站不是"妥协方案"，而是**主流方案**。

## v1 总体架构

```
                    User natural language instruction
                                  │
                                  ▼
                    ┌──────────────────────────┐
                    │  ANIMA: LLM Parser       │  Claude API
                    │  → structured TaskSpec   │
                    └──────────────┬───────────┘
                                   │
                                   ▼
                    ┌──────────────────────────┐
                    │  Grounding DINO + SAM2   │  → object detection by language
                    │  pixel → world coords    │  ("find the red block")
                    └──────────────┬───────────┘
                                   │
                                   ▼
                    ┌──────────────────────────┐
                    │  ANIMA Behavior Tree     │  py_trees
                    │  + Skill Dispatcher      │
                    └──────────────┬───────────┘
                                   │
                                   ▼
                    ┌──────────────────────────┐
                    │  ACT Policy (trained on  │  LeRobot
                    │  teleop demos) on        │
                    │  RoArm-M2-S              │
                    └──────────────┬───────────┘
                                   │
                                   ▼
                    ┌──────────────────────────┐
                    │  ANIMA Validator         │  test-and-check via vision
                    │  (DIARC-inspired)        │  → success / retry / report
                    └──────────────────────────┘
```

**架构关键点**：
- **ANIMA = 主角**：LLM-as-Parser 解析意图，py_trees 行为树编排，validator 做 test-and-check 验证（DIARC 签名特性）。
- **ACT = 低层 API**：每个原语 (`pick`, `place`, `push`) 是一个独立训练的 ACT policy，对外暴露成"函数调用"。
- **Grounding DINO + SAM2 = 感知桥梁**：负责把"红色方块"这种语言短语翻译成像素 → 世界坐标。
- **MoveIt2 兜底**：即便 ACT 训练失败，硬编码的 MoveIt2 IK 原语依然能跑 demo（Week 3 的安全网）。

### 简化后的 v1 真机系统拓扑

```
┌──────────────────────────────────┐
│  Desktop PC (Windows 11 + WSL2)  │
│  ├─ ROS 2 Humble (Ubuntu 22.04)  │
│  ├─ LeRobot + PyTorch + CUDA     │
│  │   → RTX 4090 (passthrough)    │
│  ├─ ANIMA + Claude API client    │
│  └─ MoveIt2 / py_trees           │
│         │                         │
│         │ USB (usbipd-win 转发)   │
│         ▼                         │
│  ┌─────────────────────────────┐ │
│  │  Logitech C922 Pro          │ │  ← 头顶 ~60cm 俯视
│  │  (UVC, /dev/video0)         │ │
│  └─────────────────────────────┘ │
│  ┌─────────────────────────────┐ │
│  │  RoArm-M2-S                 │ │  ← 桌沿固定
│  │  (USB serial, 自写 driver)  │ │
│  └─────────────────────────────┘ │
└──────────────────────────────────┘
```

**关键变化**：v1 真机系统**完全不需要 Pi 5、不需要 WiFi、不需要跨机通信**。所有东西插在一台 PC 上。这砍掉了原计划里**至少一周**的 multi-machine debug 时间。

### 为什么选 Logitech C922 Pro

| 因素 | 解释 |
|---|---|
| **UVC 实现成熟** | Logitech UVC 在 Linux/V4L2 上是事实标准，`v4l2-ctl --set-ctrl=exposure_auto=1 / white_balance_temperature_auto=0 / focus_auto=0` 全部按文档生效。**杂牌摄像头这些命令的行为是"看心情"，会偷偷自动调整 → 毒害 ACT 训练数据**。 |
| **ALOHA 同款** | ALOHA 论文用 C922×3，LeRobot 教程默认假设 Logitech，HuggingFace Hub 上大量项目用 C920/C922。社区 v4l2 配置脚本可直接复制粘贴。 |
| **物理稳定** | 自带三脚架孔，USB 连接器质量好，长时间工作不漂移。便宜杂牌摄像头一周内出一次问题就够你哭。 |
| **可复现性** | 数据集发布到 HuggingFace 后，别人想复现只要买同款 C922 就行——这是开源 portfolio 的隐形加分项。 |
| **架构简化** | USB 直插 PC，**不需要经过 Pi**。延迟 < 30ms，零压缩损失。 |

**不选 Pi Camera Module 3 的原因**：CSI 接口只能插 Pi → 必须经过 Pi 5 → JPEG/H264 压缩 → WiFi → PC，每帧 50-150ms 抖动 + 压缩损失。Pi Cam 自动曝光/白平衡的接口在 `libcamera`，**不在 LeRobot 工具链里**，锁起来麻烦。分辨率 12 MP 严重过剩（你只需要 ~3-5 px/mm 定位精度）。

**不选 Intel RealSense D435 的原因**：v1 是俯视固定相机 + 已知桌面平面，**像素 + 桌面 z = 唯一确定的世界坐标**——深度信息冗余。多花 $200 买一个用不到的功能不值。如果未来真的发现需要 z 信息（比如做 stack），再升级。

## 开发环境

### v1 开发环境（2026-04-07 锁定）

| 项目 | 配置 |
|---|---|
| **宿主系统** | Windows 11 |
| **开发系统** | WSL2 + Ubuntu 22.04 |
| **GPU 接入** | **CUDA passthrough** —— WSL2 通过 `/usr/lib/wsl/lib/libcuda.so` 调用 Windows NVIDIA driver，**RTX 4090 性能 100% 无损** |
| **USB 接入** | `usbipd-win` —— 把 Windows USB 设备（C922、RoArm-M2-S）转发到 WSL2 的 `/dev/video0` 和 `/dev/ttyUSB0` |
| **ROS 2 版本** | ROS 2 Humble Hawksbill (LTS, 与 LeRobot 生态最对齐) |
| **Python 版本** | 3.10 (Ubuntu 22.04 默认，与 ROS 2 Humble 对齐) |
| **机器学习栈** | PyTorch (CUDA 12.4) + LeRobot + HuggingFace transformers/datasets |
| **图形栈** | WSLg (Gazebo / RViz2 渲染稍慢但可用) |

### 为什么选 WSL2 而非双系统 Ubuntu

| 因素 | 双系统 Ubuntu | WSL2 + CUDA passthrough |
|---|---|---|
| GPU 性能（关键） | ⚠️ **本机 Linux 双系统 GPU 性能受限**（用户已确认） | ✅ 100% 满血（用 Windows driver） |
| ROS 2 Humble | ✅ 原生 | ✅ 原生 |
| LeRobot / PyTorch | ✅ 原生 | ✅ 原生 |
| 与 Windows 工具链共存 | ❌ 重启切换 | ✅ 共存 |
| 配置成本 | 1-2 小时装系统 | 30 分钟装 WSL2 + usbipd |
| Gazebo 图形 | ✅ 完美 | ⚠️ 70-85% 性能（够用） |
| USB 设备 | ✅ 直接访问 | ⚠️ 需要 usbipd 转发，每次重启需重新 attach |

**结论**：本项目 90% 的 GPU 算力消耗在 PyTorch 训练 / VLM 推理 / Grounding DINO，这部分在 WSL2 上 100% 无损。Gazebo 图形稍慢可接受。换得保留 Windows 工具链 + 不踩硬件兼容坑。

```bash
# WSL2 Ubuntu 22.04 内

echo "deb [arch=$(dpkg --print-architecture) signed-by=/usr/share/keyrings/ros-archive-keyring.gpg] http://packages.ros.org/ros2/ubuntu $(. /etc/os-release && echo $UBUNTU_CODENAME) main" | sudo tee /etc/apt/sources.list.d/ros2.list > /dev/null
sudo apt update
sudo apt install -y ros-humble-desktop python3-rosdep python3-colcon-common-extensions
echo "source /opt/ros/humble/setup.bash" >> ~/.bashrc

# v4l-utils (摄像头控制)
sudo apt install -y v4l-utils

# LeRobot
pip install lerobot
```

```powershell
# Windows: USB 转发 (需要 C922 已插上)
usbipd list
usbipd bind --busid <C922_BUSID>      # 一次性
usbipd attach --wsl --busid <C922_BUSID>  # 每次开机后
```

## 软件栈与数据流

| 模块 | 组件 | 运行位置 |
|---|---|---|
| 认知 | ANIMA LLM-as-Parser (Claude API) + py_trees + validator | WSL2 (4090 PC) |
| 感知 | Grounding DINO + SAM2 + 像素→世界坐标 | WSL2 (4090 PC) |
| 运动规划 | MoveIt2 (4-DOF, IKFast/BioIK) | WSL2 (4090 PC) |
| 学习策略 | LeRobot ACT (one model per primitive 或 multi-task) | WSL2 (4090 PC)；推理直接通过 USB 串口下发关节指令 |
| 硬件驱动 | Custom thin Python driver wrapping Waveshare serial protocol (~200 行) | WSL2 (4090 PC, USB serial 直连) |
| 数据采集 | LeRobot teleop pipeline + PDP Xbox 手柄（pygame / python-evdev） | WSL2 (4090 PC) |
| 数据集 | LeRobotDataset 格式，发布到 HuggingFace Hub | HF Hub（公共 artifact） |
| 仿真备份 | Gazebo（保留 mobile + manipulation 完整集成 demo） | WSL2 (4090 PC) |

**ROS 2 包**：复用 `arm_description`、`arm_bringup`、`anima_node`（从 `soma_home_exp_v1` 迁移并去掉 `soma_` 前缀），W2-W3 新增 `arm_perception`、`arm_manipulation`。

## ACT 技能原语

ACT 学习的原子技能，作为 ANIMA 行为树的"叶节点"。

| Primitive | 输入 | 动作 | 4-DOF 可行性 | 优先级 |
|---|---|---|---|---|
| `pick(x, y)` | 物体像素 → 世界坐标 | top-down 抓取 | ✅ | **核心** |
| `place(x, y)` | 目标世界坐标 | 在指定位置释放 | ✅ | **核心** |
| `push(from, to)` | 起点 + 终点坐标 | 非抓取式推动 | ✅ | **核心** |
| `sweep(area)` | 区域 | 多物体聚拢 | ✅ | stretch |
| `stack_on(target)` | 目标物体顶部 | 堆叠 | ⚠️ 2~3 层封顶 | stretch |

**Week 4 数据采集只采前 3 个核心原语；sweep / stack 时间允许再做。**

## 任务分级

ANIMA 在 ACT 原语之上能编排出不同复杂度的任务。任务分级直接对应 demo 视频的分镜。

### Tier 1 — 单步语言 grounding（入门 demo）
> 业内人人在做，是必须的"门票"。

- "Put the red block in the green box."
- "Pick up the apple."

### Tier 2 — 多步长时程（RT-1 / RT-2 级别）
> 显示 LLM 任务分解能力。

- "Sort the blocks by color into the matching boxes."
- "Stack the three red blocks."
- "Clean up the table."

### Tier 3 — 条件 + 空间推理（ANIMA 差异化点）
> 显示 LLM 不仅会分解，还能做条件分支与空间关系推理。

- "If there's a red block, put it in the green box. Otherwise put the blue one in."
- "Put the red block to the left of the green block."
- "Pick up the block that doesn't match the others."

### Tier 4 — Test-and-check + 失败恢复（DIARC 签名特性）
> **几乎没有 LLM-robot demo 在做这个**。这是项目的核心差异化卖点。

- "Pick up the red block."
  → 第一次尝试 → 失败
  → ANIMA 视觉验证: 物体仍在原位
  → 重试
  → 仍失败
  → 自然语言报告: "我试了两次，夹爪都滑了。要不要试试不同角度？"

## 风险与缓解

| 风险 | 缓解 |
|---|---|
| RoArm-M2-S 精度太低（hobby 舵机 ~1-2mm 重复定位） | 用更大物体（5cm 立方块），保守抓取，仅 top-down |
| RoArm-M2-S 没有官方 ROS 2 驱动 | 自写薄 Python 驱动包装串口协议，< 200 行 |
| ACT 从 teleop 数据迁移到自主执行效果差 | 采更多 demo，提升 teleop 质量，必要时 demo 时回退到硬编码原语 |
| LLM parser 产生非法 TaskSpec | ANIMA validator 捕获 + 重试；Claude 用 constrained generation |
| 物理工作区对臂的可达性不足 | Week 1 早期就验证可达性，必要时调整物体位置或臂安装方式 |
| 光照变化破坏感知 | 漫射光源、固定白平衡、不同光照下做测试 |
| 任何阶段时间滑坡 | 每周有清晰"完成"门，stretch goal 全部可砍；Week 6 不达标先砍 Week 7 的 stretch |
